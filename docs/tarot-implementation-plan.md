# 塔罗占卜项目落地实现方案

## 1. 实现目标

本项目第一阶段目标是做出一个完整可上线的塔罗占卜 MVP。核心方向：

- 不开放自由问题输入。
- 所有问题通过类别、选项和参数生成。
- 不依赖 LLM 完成基础占卜。
- 抽牌结果不做新手保护或人为改牌。
- 情绪反馈由解析文案承担，不改变牌面随机结果。
- 第一版优先做稳定、可信、可复盘的占卜体验。

MVP 必须包含：

```text
主题选择
问题类别选择
参数选择
标准化问题确认
洗牌
切牌
抽牌
翻牌
逐张牌解释
综合解析
行动建议
结果保存
历史记录
```

## 2. 推荐技术架构

如果项目是 Web 或跨端 App，建议采用以下结构。

```text
客户端
  UI 页面
  抽牌动画
  本地状态
  结果展示

业务服务
  问题类别配置
  牌库数据
  抽牌服务
  解析服务
  历史记录服务

数据层
  用户表
  占卜记录表
  牌库配置
  问题类别配置
  文案模板配置
```

MVP 可以先做成纯前端本地版本：

```text
JSON 配置 + 前端随机抽牌 + LocalStorage 保存历史
```

正式上线建议改为服务端生成正式结果：

```text
服务端生成随机种子
服务端抽牌
服务端保存结果
客户端只负责展示和动画
```

原因：如果抽牌完全在客户端执行，玩家可以通过刷新、抓包或改本地状态反复刷结果，会削弱占卜可信度。

参数只影响：

```text

推荐牌阵

结果语气
```

参数不能影响：

```text
抽到哪张牌
正逆位概率
```


### 4.6 抽牌流程

状态机：

```text
idle
-> question_confirmed
-> shuffling
-> shuffled
-> cutting
-> cut_done
-> selecting_cards
-> cards_selected
-> revealing
-> revealed
-> interpreting
-> completed
```

每个状态只允许指定操作，避免玩家跳步。

### 4.7 洗牌

输入：

```text
长按时间
拖动距离
拖动轨迹
松手时间
```

输出：

```json
{
  "shuffleDurationMs": 4200,
  "dragDistance": 680,
  "gestureHash": "9f2a1c",
  "completed": true
}
```

完成条件：

```text
长按时间 >= 1.5 秒
拖动距离 >= 300 px
或总洗牌时长 >= 4 秒
```

### 4.8 切牌

输入：

```text
切牌方向
切牌位置
置顶牌堆选择
```

输出：

```json
{
  "cutRatio": 0.43,
  "topPile": "right",
  "cutHash": "3a88d2"
}
```

切牌结果参与随机种子，但玩家不需要看到具体数值。

### 4.9 抽牌

规则：

```text
同一次占卜无放回抽牌
不重复同一张牌
每张牌独立计算正逆位
正位概率 70%
逆位概率 30%
不做新手保护
```

输入：

```text
questionId
spreadId
userId
date
shuffleHash
cutHash
selectedCardSlots
```

输出：

```json
{
  "readingId": "reading_001",
  "cards": [
    {
      "positionId": "current_state",
      "cardId": "major_06",
      "orientation": "upright"
    }
  ]
}
```

### 4.10 翻牌

规则：

```text
逐张翻开
每张牌翻开后展示牌名、正逆位、关键词
所有牌翻开后才生成综合解析
```

翻牌不改变结果，只改变展示状态。

### 4.11 解析系统

解析分为 4 层：

```text
单牌解释
牌位解释
牌组关系解释
综合判断与建议
```

MVP 不需要 LLM，使用模板和规则生成。

输入：

```json
{
  "questionId": "love_relationship_trend",
  "displayQuestion": "我和这个人在未来一个月的关系发展趋势是什么？",
  "topic": "love",
  "spreadId": "relationship_5",
  "cards": [
    {
      "positionId": "self_state",
      "cardId": "major_06",
      "orientation": "upright"
    }
  ]
}
```

输出：

```json
{
  "summaryType": "cautious_forward",
  "energyLabel": "中性偏积极",
  "clarityLabel": "仍有隐藏信息",
  "cardReadings": [],
  "combinedReading": "这组牌更接近谨慎推进...",
  "advice": "未来 7 天内适合发起一次轻量沟通...",
  "emotionalSupport": "如果你看到这个结果后仍然不安，这是正常的..."
}
```

### 4.12 历史记录

保存内容：

```text

参数
```

## 6. 解析规则实现

### 6.1 单牌解释

模板：

```text
在「{positionName}」位置出现「{cardName}{orientation}」，说明{positionMeaning}受到「{keyword1}」和「{keyword2}」影响。{topicMeaning}
```

### 6.2 综合判断

计算维度：

```text
牌面能量分
逆位数量
大阿尔卡那数量
关键牌位权重
冲突牌组合
支撑牌组合
```

示例规则：

```json
{
  "id": "cautious_forward",
  "condition": {
    "energyScore": [-1, 2],
    "reversedCount": [1, 3],
    "hasSupportCard": true
  },
  "label": "谨慎推进",
  "template": "这组牌不指向立刻完成，也不指向彻底停滞。它更像是在提醒你先处理阻力，再小步推进。"
}
```

### 6.3 情绪承接

按结果类型选择模板。

```json
{
  "resultType": "negative",
  "template": "如果这个结果让你有些失落，这是正常的。它不是在否定你的感受，而是在提醒你先把注意力放回可确认的行动上。"
}
```

### 6.4 LLM 可选接入

MVP 不接 LLM。

后续如果接入，LLM 只能做文案润色：

```text
输入结构化结果
输出更自然的长文报告
不得修改牌、正逆位、结论类型和行动建议
```

```text
牌名区域：底部预留 180 px
主体区域：中间 70% 画面
AI 生成图：不画边框、不画牌名、不画编号、不画任何文字
UI 牌框：程序统一叠加 32-48 px 边框
色彩：每张牌可有主色，但边框、牌名、编号和正逆位标记由 UI 系统统一
```

不允许：

```text
牌名文字被主体遮挡
主体过度贴边
AI 图自带边框、牌名、编号或水印
强透视导致缩略图看不清
```

### 11.2 背景规范

```text
移动端主尺寸：2160 x 3840
重要元素避开顶部 240 px 和底部 320 px
中心区域不能过亮，避免影响牌面识别
不放具体文字
可循环或可裁切
```

### 11.3 图标规范

```text
SVG 优先
线宽统一
默认单色
支持 CSS 改色
保持 24 px 小尺寸下可识别
```

### 11.4 动效贴图规范

```text
透明背景 PNG
边缘柔和
不带文字
可叠加混合
提供普通和高亮两档
```

## 17. 风险与取舍

### 17.1 最大风险：结果像随机文案

解决：

```text
牌位解释必须具体
问题类别必须明确
综合结论必须有倾向
行动建议必须可执行
```

### 17.2 最大成本：牌面美术

解决：

```text
MVP 先做 22 张大阿尔卡那
小阿尔卡那延后
缩略图自动生成
统一边框和版式降低制作成本
```

### 17.3 最大体验风险：负面结果伤害玩家

解决：

```text
不改牌
不恐吓
不绝对预言
用行动建议和情绪承接处理负面结果
```

### 17.4 最大技术风险：动画卡顿

解决：

```text
卡牌数量控制在 15 张以内
粒子效果轻量化
移动端减少阴影和模糊
优先使用 transform 和 opacity
```
